Mission

To completely implement a correct server for all of Red Hat's excellent up2date tools. Current is not expected to scale to the same mammoth proportions, but it should be fast enough and capable enough to support several hundred machines without driving the sysadmins mad.

Features

  • Production quality for 2.7 and 2.8 clients with a Red Hat 7.x/8.0 server.
  • Experimental support for 3.0 clients.
  • Full database backend in development (and working).
  • As easy to setup for the admin as we can make it.

Contributing

To discuss ideas, bug fixes, or submit patches then join the development mailing list. If your patch implements an existing RFE or fixes a known BUG, you can also/instead attach it to our Issue Tracker.

You can usually find one of us on IRC on irc.freenode.net in #current.

If you want to add a bug/rfe/whatever to the Issue Tracker without registering with tigris.org, just send them to the dev or users list. (If you are registered, you do not have to join current.) We'll add it.

Future Directions

Naturally, supporting new versions of the client and making current work on newer releases of Red Hat Linux are a priority. Other areas we look forward to:

  • Full SQL backend database:
    Development is ongoing on the trunk. The 1.5.x series will simply reimplement the features we have now, but with a much saner SQL based backend (and enough changes to sanely support it). John's spearheading this campaign.

  • Basic client management:
    Once we have the basics complete in the SQL backend (to support the same modes of use we have now) we naturally want to start supporting at least the basic client management features available in up2date. This includes full push (rhnsd/rhn_check) and being able to "group" clients together. Naturally, this has to follow the 1.5.x/1.6.x release.

  • To infinity, and beyond:
    Applet support, full HTML driven management interfaces, full errata annoucements, "proxy" server like support, there are literally dozens of really useful features in up2date that we can support, if only we write enough python. :)
